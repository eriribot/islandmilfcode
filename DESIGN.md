# islandmilfcode 手帐重构方案

## 当前问题

当前 `islandmilfcode` 已具备手帐式阅读与续写交互，但仍存在明显的模板残留与结构性问题：

- 白娅硬编码散落在类型、默认状态、提示词、通知文案、状态面板、摘要面板、变量读写逻辑中；
- `index.ts` 同时承担了入口、状态、本地动作、生成流程、楼层逻辑、变量协作等过多职责；
- 变量读写尚未抽象成统一兼容层，页面逻辑仍直接依赖当前酒馆消息变量接口；
- 当前数据结构本质上仍是“单角色模板”，无法自然承载多个攻略对象；
- 后续若直接在现状上继续加功能，返工风险很高。

## 重构目标

本轮重构目标分三层，顺序固定：

1. **手帐先模块化**：先拆职责，再继续新增能力；
2. **数据先去白娅化**：把单角色模板改成通用攻略对象结构；
3. **接入 MVU 兼容层**：让手帐在有 / 无 MVU 的环境里都能工作。

当前阶段不追求多对象总览或更大范围的 UI 改版，重点是把手帐模块变成可维护、可扩展、可迁移的基础结构。

## 数据模型方案

默认采用：`targets[] + activeTargetId`。

### 主结构

`StatusData` 目标结构应收敛为：

- `world`
- `player`
- `targets: TargetStatus[]`
- `activeTargetId: string | null`

### `TargetStatus` 最小字段

每个攻略对象最少包含：

- `id`
- `name`
- `alias?`
- `stage`
- `affinity`
- `titles`
- `outfits`
- `meta?`

### 默认规则

- 当前 UI 只展示 `activeTargetId` 对应对象；
- `activeTargetId` 不存在或失效时，回退到 `targets[0]`；
- `targets` 为空时进入无目标占位态；
- 旧 `baiya / 白娅` 数据迁移为单元素 `targets[]`；
- 旧结构只允许存在于兼容解析层，不得继续暴露给渲染、动作、提示词模块。

## MVU 兼容方案

### 目标

新增独立变量适配层，统一封装手帐运行时的数据读写。

### 兼容路径

- 若运行环境存在 `Mvu`：
  - 优先读取 `Mvu.getMvuData(...)`
  - 优先写回 `Mvu.replaceMvuData(...)` 或对应 MVU 写回接口
- 若运行环境不存在 `Mvu`：
  - 回退到现有 `getVariables / updateVariablesWith`

### 约束

- `stat_data` 继续作为主数据入口；
- 页面层不得直接调用两套底层变量 API；
- MVU / fallback 差异只能存在于适配层中；
- 无论变量来源是哪条路径，最终都必须先归一化，再进入状态与渲染层。

## 模块拆分方案

本模块后续拆分至少覆盖以下职责边界：

- `index.ts`
  - 只做入口装配、初始化、事件注册、渲染调度
- `state/store`
  - 管理 `AppState`、草稿、楼层焦点、通知、菜单状态
- `variables/adapter`
  - 管理 MVU / fallback 变量读取、归一化、写回
- `prompt`
  - 管理提示词拼装、正文清洗、楼层可见文本提取
- `actions`
  - 管理发送、流式事件、通知、回溯、重新生成
- `render`
  - 只负责状态到视图的映射与模板输出

### 约束

- 新逻辑不得继续优先堆到 `index.ts`；
- 渲染层不得直接碰变量接口；
- 动作层不得直接承担 HTML 拼接；
- 适配层不得耦合 DOM 或 prompt 文案。

## 提示词去白娅化方案

提示词层必须从“角色模板写死”改为“当前对象动态驱动”。

### 必须改掉的点

- 固定角色名；
- 固定“依存度”话术；
- 固定通知标题；
- 固定阶段文案读取路径。

### 新规则

- 提示词从 `activeTarget + world + userInput` 生成；
- 通知文案、摘要、状态标签、顶部对象信息都从当前目标读取；
- 若无 active target，则退化成通用场景续写模式；
- 回复清洗规则不能依赖固定角色名做剥离。

## UI 范围与非目标

### 当前 UI 范围

本轮只做：

- 去白娅化；
- 模块化支撑；
- active target 驱动展示；
- MVU 兼容落地。

### 当前不做

- 多对象总览；
- 对象切换器；
- 全仓 UI 大重构；
- 把手帐直接抽成全仓通用框架。

也就是说，本轮 UI 仍保持当前手帐布局，只把它从“单角色模板”改成“可承载通用对象”的结构。

## 迁移顺序

迁移顺序必须固定，避免边改边返工：

1. 类型与默认状态中性化；
2. 变量适配层抽离，并兼容旧 `baiya / 白娅` 数据；
3. Prompt 与消息格式逻辑抽离；
4. Actions 层抽离，统一管理发送 / 回溯 / 重新生成 / 流式去重；
5. Render 层改为读取 active target；
6. 最后做回归验证，确保楼层行为不退化。

## 验收标准

重构完成后，至少满足以下验收条件：

- 不再出现白娅专属硬编码业务结构；
- `index.ts` 只保留入口职责；
- `stat_data` 新结构可承载多个攻略对象；
- MVU 与 fallback 两条变量路径都能正常读写；
- user + assistant 楼层逻辑不回归；
- 楼层回溯 / 重新生成不回归；
- 重新生成成功清空 draft、失败保留 draft 的行为不回归；
- 当前手帐 UI 布局保持稳定，没有被强行扩展成多对象总览。

## 当前范围

本方案只服务于 `src/islandmilfcode` 手帐模块的后续重构，不等于全仓统一前端改造方案。

## 后续扩展点

待本轮结构稳定后，可以继续扩展：

- active target 切换入口；
- 多对象总览页；
- 更正式的 MVU schema 注册与脚本协同；
- 与 `diarytemplate` 的进一步模板抽象。

但这些扩展必须建立在本方案先完成的前提上，不能绕过模块化重构继续往入口文件叠业务。
