# islandmilfcode 手帐模块说明

## 模块目标

`islandmilfcode` 是一个手帐式的楼层阅读 / 续写界面模块。它当前聚焦于：

- 以手帐阅读器的方式浏览消息楼层；
- 在正文区域执行续写、回溯、重新生成等核心交互；
- 通过消息层变量或 MVU 兼容层读取状态并驱动 UI；
- 为后续“多攻略对象”扩展保留数据层能力。

当前范围只聚焦手帐模块本身，不负责整仓的通用 UI 基建、全局前端框架抽象或角色卡总控逻辑。

## 目录职责约定

后续重构必须以模块化为前提，不能继续把核心业务堆进 `index.ts`。推荐职责边界如下：

- `index.ts`
  - 仅负责入口装配、初始化、顶层生命周期绑定、渲染调度；
  - 不得长期承载状态机、变量读写、提示词、正文操作等核心业务实现。
- `state` / `store`
  - 管理 `AppState`、楼层焦点、草稿、通知、上下文菜单等本地状态；
  - 统一维护 state 更新入口，避免在多个文件中分散直接改写状态。
- `variables` / `adapter`
  - 统一封装消息变量读取与写回；
  - 优先兼容 MVU，回退现有酒馆变量 API；
  - 页面与动作层不得直接操作底层变量接口。
- `prompt` / `message-format`
  - 负责可见正文抽取、回复清洗、提示词拼装；
  - 不得混入 DOM 操作或变量写回逻辑。
- `actions`
  - 负责发送、流式生成、通知、回溯、重新生成等动作；
  - 统一处理成功 / 失败 / 去重 / draft 生命周期。
- `render`
  - 只负责把状态映射为界面；
  - 不直接读取底层变量，也不直接触发生成请求。

## 编程规范

### 模块化约束

- 禁止继续把核心业务持续堆进 `index.ts`。
- 单文件超过单一职责时必须拆模块。
- 新增功能优先扩展已有职责模块，而不是在入口文件临时追加逻辑。
- 手帐业务必须优先按“状态 / 变量 / 动作 / 渲染”分层。

### 命名约束

- 禁止使用角色名硬编码作为业务结构命名，如 `baiya`、`白娅状态` 一类字段名。
- 优先使用中性命名：
  - `target`
  - `targets`
  - `activeTarget`
  - `affinity`
  - `stage`
  - `world`
  - `player`
- 文案可以展示具体角色名，但数据结构、函数名、类型名必须保持通用。

### 全仓前端轻规范

本仓后续所有前端功能都应遵守下面的轻量原则：

- 入口文件只能承担入口职责；
- 页面逻辑不能长期堆成“大一统脚本”；
- 新功能落地时，优先先定职责边界，再写实现；
- 可以先局部模块化，但不能继续向“单文件继续长大”的方向演进。

## 状态与变量约定

### 状态结构约定

手帐数据层必须可承载多个攻略对象，即使当前 UI 只显示一个对象。默认采用：

- `world`
- `player`
- `targets[]`
- `activeTargetId`

默认规则：

- UI 当前只显示 `activeTargetId` 对应对象；
- 若 `activeTargetId` 无效，则退回 `targets[0]`；
- 若 `targets` 为空，则进入无目标占位态，但手帐仍需可继续工作。

### 变量来源约定

- `stat_data` 仍是主数据入口；
- 运行时优先兼容 `Mvu`；
- 无 `Mvu` 时回退现有 `getVariables / updateVariablesWith`；
- 页面层不得直接区分两套变量路径；
- 变量兼容逻辑必须收敛到适配层中统一处理。

### 兼容旧数据约定

- 旧 `baiya / 白娅` 数据必须支持迁移到单元素 `targets[]`；
- 新结构落地后，旧结构只允许出现在兼容解析层，不能继续扩散到页面、动作、类型和提示词模块。

## 提示词与文本规范

- 提示词必须从 `activeTarget + world + 当前用户输入` 动态组装；
- 禁止继续在提示词模板里写死“白娅”或其它固定角色名；
- 无 active target 时必须退化成通用场景续写模式，而不是假定某个角色一定存在；
- 通知标题、状态面板、摘要文案、顶部角色信息都必须从当前目标动态读取；
- 文本清洗逻辑不得依赖固定角色名做正则剥离。

## UI 与交互约定

### 当前 UI 范围

本轮 UI 只做“去硬编码 + 模块化支撑”，不做产品层大改版：

- 不做多对象总览；
- 不做对象切换器；
- 保持现有手帐布局与阅读方式；
- 当前仍以 active target 驱动主要展示区域。

### 必须保护的手帐核心能力

以下能力属于手帐核心交互，后续改动不得破坏：

- 楼层阅读；
- user + assistant 楼层计数逻辑；
- 回溯楼层输入；
- 重新生成该楼层；
- 重新生成成功清空 draft；
- 重新生成失败保留 draft；
- 右键楼层菜单的基本工作流。

## 后续阅读顺序

建议后续开发者按下面顺序阅读与推进：

1. 先读 `DESIGN.md`，明确重构目标、范围和迁移顺序；
2. 再看类型定义与变量适配层方案；
3. 再看状态管理与动作分层；
4. 最后再进入渲染实现与交互细节。

## 当前范围

本说明文档当前只约束 `src/islandmilfcode` 手帐模块，以及一条轻量级全仓前端模块化原则，不扩展为仓库级完整规范手册。

## 后续扩展点

未来允许在本模块继续扩展：

- 多对象切换 UI；
- 多对象总览页；
- 更正式的 MVU schema 对接；
- 通用手帐模板抽象。

但在这些扩展开始前，必须继续遵守本说明中的模块边界与命名约束。
